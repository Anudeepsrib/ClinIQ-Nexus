Integration guide

Deep Agents and Hindsight Memory in careOS

This guide explains how the platform combines deterministic routing, MCP context governance, LangGraph workflows, route-scoped Deep Agents, and governed long-term memory. It starts with the basics, then moves into the code-level contracts, safety boundaries, and advanced production considerations.

Important naming note: in this repository, MCP means the local MCPContextGovernanceService, a context policy gate that filters and transforms data before model use. Do not confuse that with the broader Model Context Protocol ecosystem unless the platform explicitly wires external MCP servers later.

Executive summary

  • Deep Agents are not routers. The deterministic intent router chooses the route first; Deep Agents are created only for registered complex routes.
  • Memory is not clinical truth. Hindsight Memory stores governed preferences or workflow continuity notes, never diagnoses, labs, medications, or treatment facts.
  • Current implementation is hybrid. The repo pins deepagents==0.5.0 and can build the official harness, but most local agent classes return deterministic structured outputs.
  • Writes are governed. A memory candidate is classified, evaluated, audited, and then stored only if approved.

01 - Basics

The four moving parts

The integration is easier to reason about if each layer has one job. Routing decides what kind of work this is. MCP governance decides what context is safe. LangGraph decides which workflow state comes next. Deep Agents and Hindsight Memory are optional depth layers used inside those boundaries.

Deterministic router

Admission control before agents

The platform routes user requests with deterministic rules and policy overrides before any Deep Agent is considered. This keeps safety-critical routing auditable and testable.

MCP governance

Context filtering and transformation

MCPContextGovernanceService.govern() enforces tenant isolation, patient scope, consent, role transformation, prompt-injection blocking, and review flags.

Deep Agents

Complex reasoning inside a route

The official deepagents package is an agent harness built on LangGraph. This repo wraps it with a route registry, scoped tools, and deterministic local fallbacks.

Hindsight Memory

Non-authoritative continuity

Hindsight Memory is careOS-specific. It stores approved, minimized preferences and workflow notes, not clinical facts or document evidence.

Rule of thumb: RAG answers "what do the governed source documents say?" Hindsight Memory answers "what safe preference or workflow pattern should the assistant remember?" Deep Agents answer "does this complex route need a planner, tool user, and critic?"

02 - Baseline architecture

Where Deep Agents and Memory sit

Deep Agents are not a top-level substitute for the platform. They are invoked after the user, tenant, route, and initial context have already been governed. Hindsight Memory is also not a model-side scratchpad; it is a governed service that persists approved records to memory_records.

03 - Deep Agents

Official harness plus careOS guardrails

LangChain describes Deep Agents as a standalone harness for complex, multi-step agents with planning, filesystem-backed context, subagents, long-term memory, and LangGraph runtime features. careOS does not hand that harness unrestricted clinical access. Instead, it wraps Deep Agent creation behind DeepAgentFactory and registers only the tools allowed for the route and role.

Registered Deep Agent routes

The current registry contains six routes, not only discharge planning and prior authorization.

Key implementation files

  • deep_agent_factory.py owns route registration and tool scoping.
  • base_deep_agent.py defines context, output contract, and optional official SDK adapter.
  • rag_tools.py re-applies MCP governance on Deep Agent retrieval.
  • memory_tools.py is the only Deep Agent interface to Hindsight Memory.
Deep Agents in this repo are a controlled capability, not a blanket autonomy layer. The factory creates an agent only after a route allowlist match, and the tool list is assembled from route and role. This application-level least-privilege model is especially important because the repository pins deepagents==0.5.0, while LangChain's newer declarative filesystem permission feature requires a later SDK version.

04 - Hindsight Memory

Governed memory is not RAG and not the Deep Agents native memory store

LangChain Deep Agents can use filesystem-backed long-term memory. careOS intentionally uses a separate service called HindsightMemoryService. That service persists database records only after classification, governance, audit logging, and write approval. This keeps memory separate from clinical source documents and easier to review.

RAG

  • Stores source-grounded clinical document chunks.
  • Requires citation metadata.
  • Used for clinical facts, document evidence, and answer grounding.
  • Filtered by tenant, patient, consent, role, and route.

Hindsight Memory

  • Stores non-authoritative preferences and workflow continuity notes.
  • Blocks forbidden clinical content such as labs, diagnoses, medications, doses, and treatment signals.
  • Retrieves only low or medium sensitivity minimized text for the exact tenant, user, role, and patient context.
  • Audits decisions with candidate hash and length instead of retaining raw blocked content.

05 - End-to-end integration

How a complex clinical workflow runs

The most important design property is re-entry into governance. Initial RAG context is governed before it reaches a workflow. If a Deep Agent needs more context, it must use a governed tool. If it identifies a safe memory candidate, the candidate must go through Hindsight Memory service before persistence.

Clinical safety

Human review is a workflow state

Discharge planning, prior authorization, safety triage, and uncertain clinical outputs set review flags. This is represented in workflow state and review task data, not only in text displayed to the user.

Memory write semantics

Candidate is not persistence

BaseDeepAgent.propose_memory_candidate() only returns a dictionary. Persistence happens only when a memory tool or API route submits that candidate to HindsightMemoryService.process_memory_candidate().

06 - Advanced controls

Safety boundaries that matter in production

The reference implementation demonstrates the core integration. A production deployment should treat each boundary below as a testable, monitorable control with explicit ownership.

Production hardening checklist

  • Move demo in-memory fallbacks to durable stores for audit, memory, and checkpoints.
  • Attach tenant, patient, sensitivity, consent, and source metadata to every retrieved chunk.
  • Add explicit memory consent, retention, expiration, deletion, and patient-visible audit policies.
  • Use a production LangGraph checkpointer instead of in-process MemorySaver.
  • Require human approval for high-risk tools and shared memory changes.
  • Measure false positives and false negatives in memory clinical-content blocking.

Upgrade path for official Deep Agents features

  • Keep DeepAgentFactory as the application-level policy gate.
  • Upgrade deepagents if you want declarative filesystem permissions.
  • Prefer read-only shared memory for organization policy and compliance material.
  • Keep Hindsight Memory as the source of governed user/workflow memory, even if the official harness uses scratch files internally.
  • Wrap custom tools with MCP and memory audit hooks because SDK filesystem permissions do not govern custom tools.

07 - Decision guide

Choose the smallest sufficient pattern

Scenario Use Why
Simple source-grounded question
Example: "summarize the latest authorized lab report."		 RAG + MCP Fast, auditable, citation-friendly, and does not need a planner.
Multi-step clinical workflow
Example: discharge readiness or prior authorization packet drafting.		 LangGraph + Deep Agent Needs explicit workflow state plus route-scoped planning, tool use, and review flags.
Safe user preference
Example: "show abnormal trends first in summaries."		 Hindsight Memory Useful across sessions, but only after non-clinical governance and audit.
Patient-facing safety concern
Example: chest pain, abnormal values, deterioration signal.		 MCP + human review Safety overrides and review state are more important than agentic depth.
Compliance or operations analysis
Example: cross-facility access anomaly or discharge bottleneck trend.		 Scoped agent + de-identification May need multi-step analysis, but clinical content must be de-identified or review-gated.

08 - Fact check notes

What was corrected or clarified

Correct: Deep Agents should be reserved for complex multi-step work, and the router should not be agentic. This matches the repo architecture and LangChain's guidance that simpler use cases can use ordinary agents or custom LangGraph workflows.
Corrected: the current Deep Agent registry has six routes: discharge_planning, chart_summary_complex, patient_message_triage, prior_authorization, hospital_operations, and compliance_review.
Corrected: Hindsight Memory is not LangChain Deep Agents' native filesystem memory. It is a careOS service that writes governed database records.
Updated: MemoryExtractor is now called inside HindsightMemoryService.process_memory_candidate(). The executed write path is extract, normalize, classify, govern, audit, write, then save.
Corrected: memory candidates produced by fallback agent classes are proposals. They are not automatically persisted unless submitted through the memory tools or memory API.
Production caution: the docs for newer Deep Agents permissions state that filesystem permissions require deepagents>=0.5.2, while this repo currently pins 0.5.0. The current least-privilege guarantee comes from careOS factory tool scoping and governed custom tools, not SDK filesystem permissions.
